## 一句话总结

提出 **DeepEncoder V2**，用 LLM 风格的因果注意力机制替代传统 CLIP 编码器，让视觉 Token 能像人类阅读一样**按语义因果顺序重排**，而非机械地从左上到右下扫描。在 OmniDocBench v1.5 上达到 **91.09%** 的综合得分（超越 Qwen3-VL-235B、Gemini-2.5 Pro 等巨型模型），且最大视觉 Token 仅 **1120** 个——是同类模型中最少的。

---

## 核心问题：为什么需要「视觉因果流」？

### 现有 VLM 的根本矛盾

传统视觉语言模型（VLM）处理图像的方式存在一个基本矛盾：

1. **图像是 2D 结构**，信息的语义关系是非线性的
2. **LLM 是 1D 因果模型**，只能处理序列化的 Token

现有方案简单粗暴地将图像 Patch 按**光栅扫描顺序**（从左上到右下）展平为 1D 序列，配合固定位置编码喂给 LLM。这对自然图像或许还凑合，但对文档来说是灾难性的：

- 报纸可能是**多栏排版**，光栅顺序会把不同栏的内容交错混合
- 表格的阅读逻辑是**行列结构**，不是线性扫描
- 公式有**嵌套层级**，需要从内到外的理解顺序

### 人类是怎么看文档的？

人眼阅读文档时遵循的是**因果驱动的语义流**：

> 想象你在看一份报纸——你不会逐像素从左上扫到右下，而是先看标题，然后跟随排版逻辑读正文，遇到图表时会跳转到相关区域，最后读图注。每一次视觉注视都**因果依赖于**前一次。

DeepSeek-OCR 2 的核心洞察就是：**让视觉编码器学会这种因果阅读顺序**。

## 架构解析：DeepEncoder V2

### 整体流水线

DeepSeek-OCR 2 的架构继承自 DeepSeek-OCR，分为**编码器**和**解码器**两部分。关键升级在编码器——从 DeepEncoder 升级到 DeepEncoder V2。

整个前向传播可以形式化为：

$$\mathbf{O} = \mathcal{D}\left(\pi_Q\left(\mathcal{T}^L\left(\mathcal{E}(\mathbf{I}) \oplus \mathbf{Q}_0; \mathbf{M}\right)\right)\right)$$

其中：
- $\mathcal{E}$：视觉分词器（Vision Tokenizer），将图像压缩为 $m$ 个视觉 Token
- $\mathbf{Q}_0$：可学习的因果流查询嵌入（$n$ 个，$n = m$）
- $\oplus$：序列拼接
- $\mathcal{T}^L$：$L$ 层带掩码注意力的 Transformer
- $\pi_Q$：投影算子，只取后 $n$ 个 Token（即因果查询输出）
- $\mathcal{D}$：LLM 解码器

### 三大核心组件

#### 1. 视觉分词器（Vision Tokenizer）

沿用 DeepEncoder 的设计：**80M 参数的 SAM-base + 两层卷积**，实现 **16 倍 Token 压缩**。

- 输出维度从 1024 降到 896（适配后续管线）
- 用窗口注意力（Window Attention）实现低成本压缩
- 参数量（80M）与 LLM 的文本嵌入层（~100M）相当

> 这一层不是必需的，可以用简单的 Patch Embedding 替代，但保留它能显著降低后续全局注意力的计算量和显存。

#### 2. LLM 风格的视觉编码器（核心创新）

这是本文**最关键的设计**。DeepEncoder V2 将原来的 CLIP ViT 替换为一个**双流注意力的 LLM 架构**：

**双流注意力机制：**

| Token 类型 | 注意力类型 | 作用 |
|-----------|-----------|------|
| 视觉 Token（前缀） | **双向注意力**（类 ViT） | 保持全局感受野，视觉 Token 之间互相可见 |
| 因果流查询（后缀） | **因果注意力**（类 LLM） | 每个查询只能看到所有视觉 Token + 前面的查询 |

**注意力掩码矩阵：**

$$M = \begin{bmatrix} \mathbf{1}_{m \times m} & \mathbf{0}_{m \times n} \\ \mathbf{1}_{n \times m} & \text{LowerTri}(n) \end{bmatrix}, \quad n = m$$

- 左上：视觉 Token 之间完全可见（双向）
- 右上：视觉 Token 看不到查询 Token
- 左下：查询 Token 可以看到所有视觉 Token
- 右下：查询 Token 之间是因果的（下三角矩阵）

**为什么用 Qwen2-0.5B 初始化？**

- 500M 参数量与 CLIP ViT（300M）在同一量级，不引入过多开销
- LLM 预训练权重本身就带有强大的序列建模能力
- Decoder-only 架构 + 前缀拼接设计被证明是关键——用 mBART 风格的 encoder-decoder + cross-attention **无法收敛**

> 作者推测收敛失败的原因是：cross-attention 将视觉 Token 隔离在单独的编码器中，导致视觉信息交互不充分。前缀设计让视觉 Token 在每一层都保持活跃。

#### 3. 因果流查询（Causal Flow Query）

因果查询的数量**等于**视觉 Token 数量：

$$n_{\text{query}} = \frac{W \times H}{16^2 \times 16}$$

采用 **Multi-Crop 策略**避免维护多组查询：

| 视图类型 | 分辨率 | 查询数量 | 裁剪数 |
|---------|--------|---------|-------|
| 全局视图 | 1024×1024 | 256 | 1 |
| 局部视图 | 768×768 | 144（共享） | 0~6 |

总输出 Token 数 = $k \times 144 + 256$，范围 **[256, 1120]**

这个上限（1120）比 DeepSeek-OCR 的 1156 还少，且与 Gemini-3 Pro 的最大视觉 Token 预算持平。

### 两级级联因果推理

DeepEncoder V2 实际上建立了**两阶段级联因果推理**：

```
阶段 1（编码器）：语义重排
  视觉 Token → 因果流查询 → 按语义逻辑重新排序

阶段 2（解码器）：任务推理  
  重排后的 Token → LLM 自回归推理 → 输出结果
```

这本质上是在探索一个全新范式：**能否用两个级联的 1D 因果推理器来实现真正的 2D 理解**。

### 解码器

解码器沿用 DeepSeek-OCR 的设计——**3B 参数的 MoE 结构**，约 500M 激活参数。本文聚焦编码器改进，解码器未做升级。

## 训练策略

三阶段训练流程，精心设计了冻结策略：

### Stage 1：编码器预训练

- **目标**：语言建模（next token prediction）
- **配置**：编码器 + 轻量解码器联合优化
- **初始化**：视觉分词器来自 DeepEncoder，LLM 编码器来自 Qwen2-0.5B-base
- **数据**：768×768 和 1024×1024 双分辨率数据加载器
- **规模**：160 × A100 GPU，batch size 640，40k iterations（约 1 亿图文对）
- **预训练后**只保留编码器参数

### Stage 2：查询增强

- **目标**：强化因果流查询的重排能力
- **配置**：冻结视觉分词器，联合优化 LLM 编码器 + LLM 解码器
- **并行**：4 级流水线并行（分词器 PP0，LLM 编码器 PP1，解码器 PP2-3）
- **规模**：160 GPU，batch size 1280，15k iterations

### Stage 3：解码器专精

- **目标**：让 LLM 更好地理解重排后的视觉 Token
- **配置**：**冻结所有 DeepEncoder V2 参数**，只更新解码器
- **加速**：同样 batch size 下训练速度翻倍以上
- **规模**：20k iterations，学习率 1e-6 → 5e-8

> 训练数据与 DeepSeek-OCR 相同（OCR 1.0 + OCR 2.0 + 通用视觉数据，OCR 占 80%），仅做了采样平衡和标签优化的微调。

## 实验结果

### 主要结果：OmniDocBench v1.5

OmniDocBench v1.5 包含 1,355 页文档，涵盖 9 大类别（杂志、学术论文、研报等），中英双语。

**DeepSeek-OCR 2 的核心优势：用最少的 Token 达到最好的效果。**

| 模型 | 最大 Token 数 | Overall ↑ | 文本 ED ↓ | 公式 CDM ↑ | 表格 TEDS ↑ | 阅读顺序 ED ↓ |
|------|-------------|-----------|----------|-----------|------------|-------------|
| GPT-4o | - | 75.02 | 0.217 | 79.70 | 67.07 | 0.148 |
| Qwen2.5-VL-72B | >6000 | 87.02 | 0.094 | 88.27 | 82.15 | 0.102 |
| Gemini-2.5 Pro | - | 88.03 | 0.075 | 85.82 | 85.71 | 0.097 |
| Qwen3-VL-235B | >6000 | 89.15 | 0.069 | 88.14 | 86.21 | 0.068 |
| PaddleOCR-VL（Pipeline） | - | 92.86 | 0.035 | 91.22 | 90.89 | 0.043 |
| **DeepSeek-OCR** | **1156** | 87.36 | 0.073 | 84.14 | 85.25 | 0.085 |
| **DeepSeek-OCR 2** | **1120** | **91.09** | **0.048** | **90.31** | **87.75** | **0.057** |

关键数字：
- **Overall +3.73%**（87.36 → 91.09）
- **公式识别 +6.17%**（84.14 → 90.31）
- **阅读顺序 ED -33%**（0.085 → 0.057），显著验证了因果流重排的有效性
- Token 数反而**减少了 36**（1156 → 1120）

### 对比同 Token 预算的模型

在相同的 ~1120 Token 预算下：

| 模型 | 最大 Token | Overall ED ↓ |
|------|----------|-------------|
| Gemini-3 Pro | 1120 | 0.115 |
| Seed-1.8 | 5120 | 0.106 |
| DeepSeek-OCR | 1156 | 0.129 |
| **DeepSeek-OCR 2** | **1120** | **0.100** |

DeepSeek-OCR 2 用最少的 Token 拿到了最低的编辑距离。

### 生产环境表现

在真实部署中，以**重复率**（Repetition Rate）作为关键可观测指标：

| 场景 | DeepSeek-OCR | DeepSeek-OCR 2 | 降幅 |
|------|-------------|---------------|------|
| 在线用户图片 | 6.25% | 4.17% | ↓2.08% |
| 预训练 PDF | 3.69% | 2.88% | ↓0.81% |

### 仍有提升空间

在 9 类文档的细粒度对比中，DeepSeek-OCR 2 在**报纸**类上表现相对较弱（文本 ED 0.139 vs OCR 1 的 0.131）。原因：
1. Token 上限对文本密集的报纸可能不够（可通过增加局部裁剪解决）
2. 训练数据中报纸样本仅 25 万，不够充分

但在**阅读顺序**指标上，DeepSeek-OCR 2 在**所有 9 类文档上全面领先**。

## 深度思考

### 1. 为什么这个工作重要？

DeepSeek-OCR 2 表面上是做 OCR 的，但它提出的「视觉因果流」思想影响深远：

**挑战了一个根深蒂固的假设**——视觉 Token 必须按空间顺序排列。这个假设在 ViT 之后几乎成了不假思索的默认选项，但它本质上是把 2D 信息硬塞进 1D 模板的权宜之计。DeepSeek-OCR 2 证明了：让编码器学会语义重排，比依赖 LLM 在固定顺序中"自行理解"要高效得多。

### 2. 两级因果推理的深层含义

论文提出的「两个级联 1D 因果推理器实现 2D 理解」是一个非常优雅的思路：

- **第一级**（编码器）：解决「看什么」和「按什么顺序看」的问题
- **第二级**（解码器）：解决「看到了什么」和「怎么表达」的问题

这种分工类似于人类视觉系统中**背侧通路（where/how）和腹侧通路（what）**的分工。

### 3. mBART 风格为什么不收敛？

论文提到用 cross-attention 的 encoder-decoder 结构无法收敛，这是一个重要的 negative result：

- Cross-attention 将视觉 Token 和查询 Token 物理隔离
- 查询 Token 只能通过 cross-attention 的"窄带"接触视觉信息
- 前缀拼接设计让视觉 Token 在每一层的 self-attention 中都保持活跃

这暗示：**多模态融合的深度（depth of fusion）比融合的方式更重要**。

### 4. 通往统一多模态编码器的路径

DeepEncoder V2 的架构天然支持扩展到其他模态：

- 共享 $W_k, W_v$ 投影、注意力机制和 FFN
- 不同模态只需各自的**可学习查询嵌入**
- 同一参数空间内可以：压缩文本、提取语音特征、重组视觉内容

这可能是通往「Native Multimodality」的务实路径。

### 5. 局限性与未来方向

- **Token 数限制**：最大 1120 对报纸等文本密集文档可能不够
- **多跳重排**：当前因果流查询数量等于视觉 Token 数量，可能需要更长的查询序列来实现多次回顾
- **通用视觉推理**：目前仅在文档 OCR 上验证，需要扩展到通用视觉理解任务
- **解码器未升级**：仍使用 3B MoE（500M 激活），编码器的提升空间可能被解码器瓶颈限制

---

## 总结

DeepSeek-OCR 2 不只是一个更好的 OCR 模型。它提出了一种全新的视觉编码范式——**让编码器像人一样"阅读"图像**，而非机械扫描。通过将 CLIP 替换为带因果注意力的 LLM 架构，实现了视觉 Token 的语义重排，在保持极低 Token 数的同时大幅提升了文档理解能力。

更重要的是，这个工作探索了一个根本性问题：**2D 空间理解能否分解为两个级联的 1D 因果推理？** 初步结果令人振奋，虽然还有很长的路要走。
